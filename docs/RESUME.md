# Mercury — 求职简历项目介绍

---

## 项目名称

**Mercury — 基于双循环架构的自我演化智能体系统**

## 一句话描述（放在项目列表第一行）

设计并实现基于 LangGraph 的自我演化智能体系统，通过「执行-反思-合成-验证」双循环架构从执行轨迹中提取可复用技能，并在安全隔离的 Docker 沙盒中执行三维门控自动回归验证，实现 Agent 能力的自主进化。

## 详细描述（面试时展开讲）

**项目背景**：目前大多数 Agent 存在“记吃不记打”和“过度消耗 Token”的问题。为了解决 Agent 无法有效内化经验、长上下文导致效率低下等痛点，设计了具有自主生成、验证并加载技能（Skill）的自我演化 Agent 系统。

**技术亮点**（按优先级排列）：

1. **执行与验证双循环架构** — 核心创新机制。设计 Executor-Evaluator-Synthesizer-Verifier 管线。Executor 执行任务并收集轨迹；Evaluator 反思踩坑点并提出技能；Synthesizer 将技能标准化为 Anthropic Agent Skills 规范 (SKILL.md)；Verifier 负责最后拦截把关。
2. **严苛的三维自动化门控（3D Gating）** — 摒弃容易产生幻觉的 LLM-as-Judge，设计基于确定性代码验收的评价标准。只有同时满足：成功率达标（Pass@1）、Token消耗下降（≤ 0.85× 基线）、交互轮次不增加（≤ 基线），并且能通过邻居任务（同类泛化）与反向触发（防误判）探测的新技能，才会被准入。
3. **安全硬隔离的沙盒机制** — 基于 Docker SDK 构建轻量化执行沙盒 `mercury-sandbox`。采用 `network=none` 断网防护、内存限制 (512MB)、去除特权 (`cap_drop=ALL`) 和超时杀进程策略，彻底隔绝 LLM 自动生成代码并执行带来的系统安全隐患与副作用。
4. **渐进式技能加载与模型路由** — 优化上下文效率：启动时仅将技能的 Manifest (Trigger + 摘要) 注入系统，只有命中 Trigger 时才全量加载具体指令内容；同时实现基于角色的 Fast/Slow 双模调度（复杂执行和验证节点用强推理大模型，轻量反思预筛用小模型），降低推理成本。
5. **确定性自动化评测管线** — 内部自建包含 CSV/JSON/Log 共 20 个数据清洗任务的基准测试集，构建端到端评测管线（`mercury bench`），实现系统演化前后 Pass@1、平均 Token、平均轮次的可视化对标与指标计算。

**技术栈**：Python 3.12 · LangGraph (StateGraph + SqliteSaver) · Docker SDK · LLM (Qwen 3.6 Plus/Flash) · Pytest (118+ Passed Tests) · Typer CLI

**项目规模**：端到端评测闭环实现，完整的项目 CI/CD 结构和测试覆盖率支持。

## Agent 架构详解

### 双循环演化图

```text
            ┌──────────┐
       ┌───▶│ executor │───┐
   loop│    └──────────┘   │ done / timeout
       │                   ▼
       │              ┌──────────┐    propose      ┌─────────────┐    write     ┌──────────┐
       └──────────────│evaluator │────────────────▶│ synthesizer │─────────────▶│ verifier │
                      └──────────┘                 └─────────────┘              └────┬─────┘
                            │ skip                                                   │
                            ▼                                                        ▼
                           END                                              status: verified | rejected
```

### 核心设计模式

1. **闭环验证（Closed-Loop Verification）**：技能提取不是“生成即用”，必须在 Verifier 内经历原任务重放、同组相邻任务（Neighbour）泛化探测、以及异组任务（Anti-trigger）防误触发测试。
2. **渐进式披露（Progressive Disclosure）**：防止“知识过载”，禁止将所有历史经验直接塞进 Prompt。所有技能只向 Agent 暴露简短描述，Agent 需调用 `load_skill` 工具按需获取详细避坑指南。
3. **状态溯源与持久化**：使用 SqliteSaver 对基于图（StateGraph）的每一步状态切换进行持久化打点，确保长链路调用断点可恢复，也方便截取完整 Trace 给 Evaluator 进行诊断分析。

## 面试高频问题预判

| 问题 | 回答要点 |
|------|----------|
| 为什么要设计一套专门的 Verifier（验证器）和门控？ | 大模型总结的经验常常是次优或带有幻觉的。没有 Verifier，错误经验的积累会导致 Agent 陷入退化螺旋（污染知识库）。我们的三维门控强制新技能必须能带来量化的正向经济收益（降 Token 降轮次）且能向外泛化。 |
| 三维门控中为什么要限制 Token ≤ 0.85× 基线？ | 加载一个外部技能文件本身就会消耗 1 轮交互和额外的输入 Token。如果引入的新技能无法大幅度消除 Agent 的试错与盲目推理，那么它在经济性上就是亏本的。85% 是一道硬阈值，防止系统逐渐变得臃肿。 |
| Agent 的沙盒安全隔离是怎么做的？ | 使用 Docker SDK 分配短生命周期容器。强制 `network=none` 切断对公网和内网的其他访问，剥夺所有特权（cap_drop=ALL），约束极小的内存，并用 GNU timeout 阻断死循环。这是防范 LLM 不受控操作的关键基石。 |
| Evaluator 触发反思的时机是什么？ | 按需反思。只有任务失败，或者 LLM 交互轮次过多（比如 ≥4 轮，说明踩坑且试错了）才会触发。对于“一发入魂”的简单成功（1~2轮即完成），会直接跳过反思，从而避免产生没有意义的垃圾总结。 |
| 这个系统和直接使用 LangChain 有什么不同？ | LangChain 只是 API 的胶水。本项目是一个更高维度的“产生技能元系统”，LangGraph 在这里发挥的是复杂流程图编排（DAG）和长状态打点保存的作用，我们的核心工程难点在演化机制（如何总结技能、验证泛化性、避免提示污染）。 |

## 面试深度问题解答（项目介绍及展望）

### 1. 为什么坚决不用 LLM-as-Judge 进行演化验证，而坚持纯代码的验证（Deterministic Eval）？
- **项目介绍**：在 Agent 自主演化闭环中，如果用大模型（哪怕是更强的大模型）去评判另一个模型提取的技能是否优秀，极易产生“盲目包容”或“自圆其说”的幻觉闭环（也就是不管多烂的代码，只要没报错大模型都有可能判通过）。本项目面对的是数据清洗域，客观上有硬性的标准答案，因此我采用代码级断言校验，保证成功率（Pass@1）是100%可信的硬指标。
- **改进与展望**：虽然目前保证了绝对置信，但限制了演化系统的应用域（目前限定在格式确定领域）。如果未来拓展到创意类、投资策略等开放域，可以引入一套“复合检验体系”（规则断言 + 数据分布检验 + LLM 判断），用多维度量化得分取代单纯的代码 bool 值。

### 2. 你们是如何合成新技能（SKILL.md）的？这套文件格式有什么讲究？
- **项目介绍**：我们通过 Evaluator 获取完整的 `TraceStep`，反思出核心避坑指南后，交给 Synthesizer 输出一套符合 Anthropic Agent Skills 推荐格式的 `SKILL.md`。其分为两部分：
  1. 上方的 YAML Frontmatter（包含 Trigger 词、适用场景标签、以及提取时的 Baseline Metrics 数据）。
  2. 下方的 Markdown 正文（包含具体的避坑策略和解决步骤）。
- **优化点**：这种结构的分离，是实现“渐进式披露”的前提：系统启动时，仅仅扫描 YAML 将 Trigger 装载给大模型，极大降低了上下文负担；只有当大模型自主认为匹配触发词，调用 `load_skill` 时，才会将包含长篇文字的 Markdown 正文送入推理流。

### 3. 如果演化生成的技能，反而误导了其他本没有问题的任务（即误触发/知识污染），怎么解决？
- **项目介绍**：在构建验证管线时，我们专门在 Verifier 里加入了 **反向触发探测（Anti-trigger）** 机制。每次考核新技能时，除了在原始任务上跑，还会选取一个非同类的异组任务（比如一个纯粹的 JSON 解析任务）。如果在异组任务的轨迹中，发现 Agent 被宽泛的 Trigger 词诱导去调用了这个毫不相干的技能，说明该技能描述“过宽”造成了污染，Verifier 会立刻将其拒绝归档。

### 4. 项目中的状态流转（AgentState）和历史轨迹具体是如何被利用的？
- **项目介绍**：依托 LangGraph，我们的 `AgentState` 中不仅有基础的对话 `messages`，更加入了详尽的执行流监控（如 `traces`，包含每一步的大模型输出摘要与 Python 沙盒真实执行结果）以及门控判定（`verification_outcome`）。当流程到达 Evaluator 时，不需要将冗长无比的 message 全部喂入，而是直接将其压缩提纯过的 `traces` 喂给模型进行总结。由于引入了检查点持久化机制，每一个状态切片都可以轻松地回档。

### 5. 针对项目未来优化，如何处理因为大模型能力变强导致的“演化失速”？
- **项目介绍**：在后期的开发实测中，我发现如果采用能力极强的大模型（比如原生的 Qwen-Plus 最新版），模型经常能 2~3 轮不踩坑直接解决 baseline 任务。这就导致给新技能留下的“优化空间（Headroom）”变得极小。如果新技能无法证明能突破 0.85× 的 Token 压缩阈值，就会触发退化拦截导致无法生成演化。
- **改进与展望**：
  1. **任务分级与难度递增（Curriculum Learning）**：持续引入更刁钻、更容易诱发幻觉的评估集，从而人为拉开基线模型的性能梯队。
  2. **提纯与知识折叠（Memory Consolidation）**：当零散的领域小技能堆积过多，系统应该触发定期的“折叠归约”，把数个细粒度的同类技能重新提炼合并为一条大准则，从而减小频繁 `load_skill` 带来的轮次磨损。
