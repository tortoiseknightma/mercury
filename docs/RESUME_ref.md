# StockAgent — 求职简历项目介绍

---

## 项目名称

**StockAgent — AI 驱动的自主股票投资 Agent**

## 一句话描述（放在项目列表第一行）

自主设计并实现了一个连接券商账户的 AI 投资 Agent，集成多维分析引擎、三层记忆系统、LLM 辩论机制和风控体系，支持从信号生成到自动交易的完整闭环。

## 详细描述（面试时展开讲）

**项目背景**：为解决个人投资中信息过载、情绪化交易、缺乏系统性风控的问题，设计了一个自主运行的投资 Agent，能够持续监控持仓、分析市场、执行交易。

**技术亮点**（按优先级排列）：

1. **12-Agent 多 Agent 协作架构** — 核心差异化特性。设计了四层协作系统：分析师层（4个并行分析师）、研究辩论层（Bull/Bear 辩论 + ResearchManager 裁决）、执行层（Trader 生成交易提案）、风控辩论层（激进/保守/中性三方辩论 + PortfolioManager 最终裁决）。所有 Agent 通过共享状态（AgentState）通信，支持辩论驱动的决策流程。

2. **轻量级状态图编排器（TradingGraph）** — 自研轻量编排引擎，实现分析师并行执行、辩论轮次控制、条件路由。采用共享状态模式避免 Agent 间直接调用，支持 fast/slow LLM 双模策略（分析师用快速模型，裁判用深度思考模型）。

3. **情境记忆与反思系统** — 每个辩论角色（bull/bear/aggressive/conservative/neutral）拥有独立的情境记忆（SituationMemory），基于 SQLite FTS5 全文检索历史教训。交易后反思引擎（AgentReflector）自动为每个角色提炼可复用教训。

4. **LLM 驱动的分析模块群** — 6 个 LLM 模块：财报解读（EPS/营收/指引）、新闻深度分析（60/40 混合评分）、投资论点生成（bull/bear case + 催化剂）、风险评估（组合级风险打分）、反思记忆引擎（交易后自动提炼教训存入长期记忆）。

5. **三层持久记忆系统** — 长期记忆（每轮推理注入，防止重复犯错）、研报语料库（FTS5 全文检索，按需查询）、交易决策日志（含完整上下文和结果追踪），支持跨会话学习与策略复盘。

**技术栈**：Python 3.11 · SQLite FTS5 · LLM 多厂商客户端 · ib_insync · yfinance · YAML 配置 · 多渠道通知（飞书/Telegram/CLI）

**项目规模**：~17,300 行 Python 代码，493 测试函数，CI 流水线

## Agent 架构详解

### 四层 Agent 协作系统

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1: 分析师层（并行执行）                            │
│  ┌───────────┐ ┌───────────┐ ┌──────────┐ ┌──────────┐ │
│  │Technical  │ │Fundamental│ │Sentiment │ │ Macro    │ │
│  │Analyst    │ │ Analyst   │ │ Analyst  │ │ Analyst  │ │
│  └───────────┘ └───────────┘ └──────────┘ └──────────┘ │
└─────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────┐
│  Layer 2: 研究辩论层（辩论驱动）                          │
│  BullResearcher ←→ BearResearcher → ResearchManager     │
│  （N 轮交替辩论）          （裁决生成投资计划）            │
└─────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────┐
│  Layer 3: 执行层                                         │
│  Trader → 生成具体交易方案（标的、方向、数量、时机）        │
└─────────────────────────────────────────────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────┐
│  Layer 4: 风控辩论层（三方辩论）                          │
│  Aggressive ←→ Conservative ←→ Neutral → PortfolioManager│
│  （M 轮三方交替）          （最终裁决 BUY/HOLD/SELL）     │
└─────────────────────────────────────────────────────────┘
```

### 核心设计模式

1. **共享状态通信**：所有 Agent 通过 `AgentState` 数据结构通信，无直接调用，支持审计和调试
2. **辩论驱动决策**：关键决策（多空判断、风控评估）必须经过对抗性辩论，避免单一视角偏差
3. **记忆前置于推理**：每个决策角色在推理前检索相关历史教训，实现经验积累
4. **渐进式架构**：保留现有规则引擎作为基线，LLM 进行深度解读和辩论

### 关键技术实现

- **TradingGraph 编排器**：轻量级状态图引擎，控制 Agent 执行顺序和辩论轮次
- **SituationMemory**：基于 SQLite FTS5 的情境记忆，支持每个角色独立记忆空间
- **AgentReflector**：交易后自动为每个角色提炼可复用教训，实现持续学习
- **双 LLM 策略**：分析师用 fast_model（快速分析），裁判用 deep_model（深度思考）

## 面试高频问题预判

| 问题 | 回答要点 |
|------|----------|
| 为什么设计 12 个 Agent 而不是单个 LLM？ | 单一 LLM 容易产生确认偏差。多 Agent 通过专业分工和对抗性辩论，模拟真实投资团队的决策流程，提高决策质量。 |
| Agent 间如何通信？为什么不用直接调用？ | 通过共享状态（AgentState）通信，避免紧耦合。好处：1）可审计每个 Agent 的推理链 2）支持辩论轮次控制 3）便于调试和测试。 |
| 辩论机制有什么好处？ | 强制 LLM 从对立面思考，避免单一视角偏差。类似真实投委会的对抗性讨论，裁判裁决比单一输出更可靠。 |
| 情境记忆如何工作？ | 每个辩论角色有独立记忆空间，基于 SQLite FTS5 检索历史教训。交易后反思引擎自动提炼可复用教训，实现经验积累。 |
| 和量化交易有什么区别？ | 这是 Agent 架构（有记忆、有推理、有复盘、有辩论），不是纯信号回测。重点在 agent 设计和决策流程优化，而非 alpha 发现。 |
| 为什么不直接用 LangGraph？ | 依赖重且需求简单。自研轻量编排器满足需求，避免引入重量级框架，同时保持对编排逻辑的完全控制。 |

## 面试深度问题解答（项目介绍及展望）

### 1. 具体如何在多个Agent之间共享状态，对什么数据结构进行读写？
- **项目介绍**：本项目通过一个名为 `AgentState` 的数据类（DataClass）实现状态共享。这是一种中心化状态管理的模式。所有的 Agent 不直接相互调用或传递消息，而是统一通过读取和修改这个局部的 `AgentState` 实例流转数据。这保证了各层级依赖解耦，也便于追踪审计每个 Agent 的推理链。
- **改进与展望**：当前由于是单机单条主线程上的状态流转，直接使用内存实例没有问题。如果未来扩展到分布式高并发场景（如同时处理上千只股票的并发分析），可以将其抽象为事件溯源架构（Event Sourcing）或者使用 Redis 等高速内存数据库管理远程共享状态。

### 2. 分析师的Agent角色具体如何去操作？
- **项目介绍**：分析师层（技术、基本面、情绪、宏观四类分析师）采用并行执行的方式。具体操作为：(1) 数据获取：获取专属于该领域的底层数据；(2) 基线构建：调用原有规则代码提取基础面貌判断（防范纯大模型幻觉）；(3) 提示词合成：将数据、基线信号与目标封装进 Prompt；(4) LLM 推理：调用轻量级模型快速提炼见解；(5) 产生标准报告：输出写回 `AgentState` 中相应的独立字段。
- **改进与展望**：未来可演进为支持 Tool Calling（工具调用）模式的自治智能体。由分析师主动根据市场情况决策调用何种接口（如舆情分析师主动请求查询推特最新动向），更动态地扩展能力边界。

### 3. 为什么不考虑使用现有的Agent框架？
- **项目介绍**：不采用 LangGraph 或 AutoGen 等主流框架，主要是出于**避免过度工程与保证可控性**的考量。项目当前的执行逻辑是在已有系统上叠加包裹（渐进式升级），呈现一个路径固定的带分支图结构（几次辩论控制）。自研轻量的 `TradingGraph` 编排引擎使用简单的 Python 类即可完成循环管控，这避免了重型框架引入的复杂依赖和学习成本。
- **改进与展望**：随着多 Agent 逻辑更趋复杂（例如需要不确定次数的回调自修复节点、或发展出不同大群进行动态交互），重型框架的规范性与开源生态（兼容 LangChain 任意大模型与外部内存组件）才会体现巨大价值。那是未来彻底重构时可探索的方向。

### 4. 如何保证多个Agent并行时，同时对同一个AgentState读写，不冲突？
- **项目介绍**：当前并行操作主要集中在第一阶段的四个分析师并发分析。架构上采用**职责隔离**从源头避免了冲突竞争：分析师虽然在并发调度（如线程池或异步执行）下运行，但只会写入 `AgentState` 中专属于自己的属性树叶节点（如只能赋值给 `technical_report` 和 `fundamental_report` 字段）。并在所有分析师返回后，由 `TradingGraph` 统筹完成一次原子的集中汇聚。
- **改进与展望**：当系统发展为持久化高并发更新模型时，可以通过类似乐观并发控制（Optimistic Concurrency Control）引入自增版本号，或者转为单向事件流累加（Append-Only Events）再通过 Reducer 来重构 State，确保更新绝对安全。

### 5. 为了保证Agent系统崩溃后能够从断点继续，应当如何去优化整个系统？
- **项目介绍**：目前此 Agent 开发阶段都是执行单次分析，状态都只保存在运行时内存里。一旦执行到后半段（如风控辩论阶段）因为网络掉线导致进程崩溃，就会被迫从头再次重发海量的 LLM 请求。
- **改进与展望**：最核心的改善就是建立 **Checkpoint（检查点）机制**。需要在编排引擎节点切换的流转间隙，将当下的 `AgentState` 以及进行到的进度指令转化为 JSON 并持久化到本地存储或者云数据库存储方案里；等服务自我拉起后，反序列化最后一次的 Checkpoint 重新下发驱动。

### 6. 对于Agent记忆系统的选型，如何在SQLite、Redis、ProgreSQL中做选择？
- **项目介绍**：在目前的落地开发中，我们选择了最内聚的 **SQLite (搭配其 FTS5 全文搜索)**。它的特点是非常好地契合了“轻量、快速单机起步”要求，无配置安装就实现了简单的长短教训全文抽取和复盘留档支持。
- **改进与展望**：长期来看：
  - **Redis** 应聚焦于极高频瞬发的状态路由数据存查与缓冲通信。
  - **PostgreSQL 搭配 pgvector** 将是整个长期知识库与持久记忆管理的核心走向。由于单纯基于 BM25 关键词容易查非所用，升级到成熟关系型数据库加本地向量计算（或者直接接入专用企业向量库）才能完成在海量金融数据中的“记忆特征对齐检索”。

### 7. 这个项目中具体怎么定义AgentState？
- **项目介绍**：项目中的 `AgentState` 是由 Python 的 `@dataclass` 声明而成的结构体。它大块拆解为了五个逻辑分区：(1) **公共输入区**（保存 ticker / 时间）；(2) **各并行的结果落地区**（存放专业报告）；(3) **旧模块产物承接区**（融合原有规则跑出的打分）；(4) **多轮辩论追踪区**（分别存放两类辩论的会话上下文、上一个发言归属者和循环计数器）；(5) **最终拍板及决策日志区**。通过严格声明这些字段管理整个工作流运转生命周期。
- **改进与展望**：后续重构时建议将 `AgentState` 挂载到具有类型强校验的方案（比如 `Pydantic models`）上，不仅支持强制序列化/反序列化，当大模型生成数据 Schema 不符时还能触发 Validation Error 进入自动的反省重试（Self-Reflection/Correction）逻辑。

### 8. 项目中有提到三层持久记忆系统，具体如何实现，优点是什么？
- **项目介绍**：这三层分别是 (1)以 SQLite 等保存教训摘要的**角色情境记忆**；(2)供全文检索深度历史的**研报与分析纪要的语料记忆引擎**；(3)供事后归因溯源提供整量数据的**原交易流水归约记忆**。
  在业务层里，核心在于流程闭环：Agent 做出决策生成新观点前必须要经过一次 Retrieve (RAG 操作检索经验)，在交易之后一定会进入 `AgentReflector` 发起一次 Reflect (反思新教训并注入记忆层)。
  **优点**：大幅缩小冗余的长提示词 token 消耗（替代直接塞入海量日志），从底层解决基座大模型“记吃不记打”重复犯错的问题，也赋予了不同微调视角（看多/看空）越来越个性化的知识积淀体系。
- **改进与展望**：在架构上可以深挖“学习与遗忘并存机制”，加入**记忆折旧忘却模型（Forgetting Curve）**——使落后市场特征很久的早期死板教训平滑降权被遗弃；并在后台设计定时激活的**元提纯（Memory Consolidation）**，用具备长上下文能力的大模型周期性对散碎经验进行提纯，萃炼成为高浓度的交易准则纪要。


### 9. 关于“辩论机制”的具体实现逻辑和终止条件是什么？
- **项目介绍**：辩论并非是通过大模型自由无限对话实现，而是被严格编排在 `TradingGraph` 内。它使用 `for _ in range(max_rounds)` 来进行固定轮次的控制。在辩论过程中，各参与 Agent 读取并追加 `AgentState` 中的 `DebateState`（包含 rounds、count、last_speaker 等追踪字段）。最后交由 Manager（裁判 Agent）进行汇总裁决。
- **改进与展望**：目前的“固定轮次”比较僵化。改进建议是引入**动态终止机制（Dynamic Termination）**，让裁判 Agent 在每一轮结束后评估双方共识程度；或者采用基于多维打分（如逻辑严密性、证据充分性）的量化指标，当分数达到阈值时提前结束辩论，以节省 Token 成本和时间。

### 10. 你的项目中使用了 fast/slow LLM 双模策略，具体是如何分配和管理的？
- **项目介绍**：在实例化 `TradingGraph` 时，显式注入了 `fast_llm` 和 `deep_llm`（即 llm 客户端）两个实例。在 Agent 分配上遵循“干活的用小模型，拍板的用大模型”：基础的四类分析师（如 Technical, Fundamental）、辩论角色（Bull, Bear）以及生成交易提案的 Trader 都使用速度快、成本低的 `fast_llm`；而负责最终决策的 `ResearchManager` 和 `PortfolioManager` 则使用具备更强推理能力的深度思考模型 `deep_llm`。
- **改进与展望**：未来可以开发**动态模型路由（Dynamic Model Routing）**模块。当市场平稳或输入数据较简单时，全部走小模型；当监测到 VIX 指数飙升或财报季数据极度复杂时，自动升配为全链路大模型。

### 11. SQLite FTS5 的全文搜索在“情境记忆”中具体怎么实现的？遇到了哪些技术难点？
- **项目介绍**：通过在 SQLite 中创建一张 `situations` 物理表和一张 `situations_fts USING fts5` 虚拟表同步数据。记忆检索时，将当前市场情况作为 query 传入，利用 FTS5 的 `MATCH` 语法检索相关历史教训。
- **遇到的难点与解决**：FTS5 解析器对特殊符号（如 `-`, `:`, `*`, `"`）极为敏感，未经处理的自然语言用户态文本会直接引发 SQL 语法报错。我们的解决方案是内部封装了一个 `_fts_query` 静态方法，对查询进行清洗和转义，并将 token 通过 `OR` 拼接，保证了高鲁棒性的模糊匹配。
- **改进与展望**：纯基于关键词的全文搜索对同义词（如“加息”与“利率上升”）束手无策。下一步的演进方案是引入**混合检索（Hybrid Search）**，即保留 BM25 关键词匹配的同时，引入 `pgvector` 或 ChromaDB 增加基于 Embedding 的语义向量检索，从而大幅提升历史教训的召回准确率。

### 12. 架构中提到的“保留现有规则引擎作为基线”，具体是怎么和 LLM 结合的？
- **项目介绍**：我们防范大模型“计算幻觉”（如算错均线、PE）的方法是**规则前置**。系统首先运行纯 Python 编写的规则分析器（如 `TechnicalAnalyzer`），生成像 `TechnicalSignal`、`FundamentalSignal` 这样的强类型对象。这些包含硬计算结果的数据，会作为 Context（基线数据）被包裹进提示词中提交给分析师 Agent。LLM 的职责被严格限定在“对这些绝对正确的数据进行趋势解读和因果分析”，而非从头计算。
- **改进与展望**：未来可以将这种模式固化为**“数据预处理节点 -> 洞察节点”的标准 Pipeline**，并在输出端使用强类型校验框架（如 Pydantic）配合 JSON Schema 来验证 LLM 生成的报告格式，甚至在内容上通过交叉验证确保其不与基线信号自相矛盾。

### 13. 这个项目在生产环境是如何部署的？体现了哪些工程化的考量？
- **项目介绍**：项目部署于阿里云服务器，采用了基于 `Docker Compose` 的多容器架构（Nginx 反向代理 + FastAPI 后端应用 + 无头 IB Gateway）。核心的工程考量包括：
  1. **资源与稳定性优化**：摒弃了带有笨重 GUI 的 TWS 桌面端，改用 `ghcr.io/unusualalpha/ib-gateway` 官方 Headless 镜像。不仅将内存占用压缩到约 500MB，还支持断线自动重连，满足 24 小时无人值守的自动化交易需求。
  2. **复杂网络环境处理**：针对国内服务器访问海外数据源（如 Yahoo Finance）被墙的问题，在容器层面注入了 GFW 代理配置（`HTTP_PROXY`，如通过 `host.docker.internal` 穿透宿主机代理），确保全球市场数据获取的稳定性。
  3. **可维护性与容灾**：配置了 `restart: unless-stopped` 实现宿主机重启后自动拉起服务；使用 Docker json-file 驱动设定日志轮转（`max-size=10m`），防止服务日志撑爆磁盘；通过 Telegram Bot API 实现了交易行为和系统异常的实时告警通知。
- **改进与展望**：当前仍是单节点 Docker Compose 部署。如果未来要同时承载大量用户或高频并发，可迁移至 Kubernetes (K8s) 集群进行弹性扩缩容，并引入 Prometheus + Grafana 的立体化监控体系，替代目前简单的 API 状态端点检查。
